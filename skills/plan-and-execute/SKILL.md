# 规划与执行 (Plan and Execute)

## 概述

本技能定义通用的「规划-确认-执行-验证」工作流，处理复杂、多步骤且具有一定风险的任务。通过强制性的规划阶段和用户确认环节，减少理解偏差，确保每一步改动可控且可验证。

## 何时使用

- **复杂产品需求**：涉及多个功能点、前后端协作、数据库变更或第三方服务集成。
- **技术重构/迁移**：如拆分大文件、迁移旧接口、重构核心逻辑等。
- **任务需要分步实施**：每步之间存在依赖或需要独立验证。
- **用户要求**：「制定计划」、「分步执行」或任务本身复杂度较高。

## 核心流程

### 1. 规划阶段 (Planning)

在开始任何代码改动前，必须先创建或更新**计划文件**（通常为 Markdown 格式）。

- **计划文件路径**：建议放在项目文档目录或与改动模块同级，如 `docs/feature-plan.md` 或 `xxx/SPLIT_PLAN.md`。
- **先澄清后规划**：若发现目标定义不清、验收标准缺失、实现边界不明确或存在多种实现歧义，必须先向用户提出具体澄清问题，拿到明确答复后再继续完善计划。
- **禁止猜测指定计划**：在需求信息不完整或存在歧义时，严禁代理基于主观判断直接指定计划、默认方案或擅自确定步骤优先级。
- **必要内容**：
  - **目标**：清晰描述最终要实现的功能或达成的状态。
  - **方案设计（针对产品需求）**：简述实现思路、涉及的模型变更、API 变动等。
  - **任务列表/步骤**：将大任务拆分为具体的、可操作的单步（Step 1, 2, ...）。
  - **每步验证方式**：明确每步完成后如何证明其正确性（如单测、lint、冒烟、数据对比、UI 检查）。
  - **状态跟踪**：记录每步的状态（未开始、进行中、已完成）及验证结果。
  - **PR/Issue 引用格式**：统一使用 `owner/repository#issue_number`，便于跨仓库追溯。

### 2. 确认阶段 (Confirmation)

将计划要点呈现给用户，**严禁在此阶段直接修改业务代码**。

- 向用户简述计划的目标、关键步骤和验证手段。
- 明确询问用户是否同意该计划，或是否有调整建议。
- **只有在用户明确表示「同意」、「开始执行」或「执行第 N 步」后，方可进入执行阶段。**

### 3. 执行与推进 (Execution & Progression)

严格按照计划分步实施，遵循「单步执行」原则。

- **单步原则**：一次只执行计划中的一个步骤（或一组强相关的原子改动）。
- **状态同步**：开始执行某一步前，先在计划文件中将该 Step 标记为 `进行中 (In Progress)`。
- **完成标志**：每步完成的最终标志通常是相应的 Pull Request 被合并；或用户明确指示该步已达成目标后，标记为 `已完成 (Completed)`。
- **反馈循环**：
  - 每步代码改动完成后，立即进行验证（见「验证阶段」）。
  - 验证通过后，提交 PR；在计划文件中记录 PR 链接及验证结果。
  - 向用户汇报进度，推进到下一步。
- **变更管理**：
  - **计划调整**：若发现原计划有误或需调整，必须**先更新计划文件**并告知用户原因，获得确认后再继续。
  - **意外中断**：若任务因报错或环境问题中断，应在计划文件中记录当前断点和问题，修复后再恢复执行。
- **改动最小化**：每步只做与该步骤相关的改动，避免顺手修改无关代码。
- **子代理使用**：必要时可调用子代理（如 `mcp_task`）执行具体子任务：
  - **适用场景**：单步涉及大量代码探索、跨文件搜索、多仓库并行操作等。
  - **主代理职责**：保持对整体计划、步骤状态和进度的掌控；根据子代理返回结果更新计划文件并推进下一步。

### 4. 验证阶段 (Verification)

每步完成后，必须立即进行验证。

- 按照计划中定义的验证方式执行（如 `pytest`、`ruff check`、浏览器测试等）。
- **验证失败处理**：若验证不通过，必须在本步内修复，直到验证通过。严禁带病进入下一步。
- **记录结果**：将验证结果更新到计划文件中。

### 5. 迭代与完成 (Iteration & Completion)

- 更新计划文件中的步骤状态。
- 向用户反馈当前进度，并询问是否继续下一步。
- 全部完成后，在计划文件中标记「任务完成」并记录日期。

## 核心原则

- **先规划，后动手**：没有计划，不动代码。
- **先澄清，后细化**：需求或实现存在歧义时，先提问确认，不凭假设推进。
- **不明确不拍板**：未获得用户明确输入前，不得替用户做需求取舍或指定执行计划。
- **先确认，后执行**：没有用户确认，不动代码。
- **一步一验证**：每步必验证，验证必通过。
- **计划即文档**：计划文件是任务的唯一事实来源，随进度实时更新。
- **必要时委托子代理**：复杂单步可交给子代理执行，主代理专注规划与进度控制。

## 计划文件示例结构

```markdown
# [需求名称] 开发计划

## 目标
- [ ] 实现功能 A
- [ ] 优化逻辑 B

## 方案设计
- 数据库、API、模型等变更说明

## 任务列表
| 步骤 | 内容 | 验证方式 | PR 链接 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 步骤 1 描述 | 单测/lint/冒烟 | owner/repo#123 | 已完成 |
| 2 | 步骤 2 描述 | 验证方式 | - | 进行中 |
| 3 | 步骤 3 描述 | 验证方式 | - | 未开始 |

## 执行与验证记录
- **Step 1**: [日期] 验证通过。PR: owner/repo#123
- **Step 2**: [日期] 进行中。
```
